# React 中什么情况下不应该用 `memo`

`React.memo` 不是默认优化项，而是一个有明确边界的性能手段。

很多团队一开始会把它当成“函数组件的标准外壳”，看到组件就包一层 `memo`，再配套一堆 `useMemo`、`useCallback`。表面上像是在做优化，实际上常常是在把代码复杂度往上抬。

从 React 底层实现看，`memo` 的作用不是“禁止重渲染”，而是“在满足条件时跳过这次子树工作”。它依赖浅比较、依赖引用稳定性，也依赖组件本身没有被 state、context、调度更新打断。只要理解这一点，就会发现：很多场景下它要么没收益，要么收益很小，要么代价更高。

这篇文章只回答一个问题：**什么情况下不应该用 `memo`**。

## 一、先把底层机制说清楚

`React.memo` 的核心不是缓存 JSX，而是给函数组件加了一层“是否可以复用上一次结果”的判断。

React 进入这个组件对应的 fiber 时，会先看一组条件：

- 这次 props 和上次 props 是否一致
- 组件自身有没有 state 更新
- 有没有 context 变化
- 这条路径上有没有必须继续向下处理的更新

如果这些条件都允许，React 就可以在 `beginWork` 里做 bailout，跳过这个组件函数的执行，尽量复用上一次的子树。

这意味着两件事：

- `memo` 只影响“这个组件这次能不能跳过”
- `memo` 不会阻止 state、context、外部订阅、父级调度带来的真实更新

所以，`memo` 的本质不是“让 React 不更新”，而是“减少无意义的重复计算”。

## 二、`memo` 省掉的是什么

它省掉的主要是：

- 函数组件本身的执行
- JSX 生成
- 这棵子树这次的协调工作

它带来的成本也很明确：

- props 比较成本
- 额外的边界判断
- 对引用稳定性的依赖
- 为了配合它而引入的 `useMemo` 和 `useCallback`

所以 `memo` 不是白赚的，它是一个交换。

只有当“省掉的渲染成本”明显大于“比较成本 + 维护成本”时，它才值得用。

## 三、哪些场景不应该用 `memo`

### 1. 组件本身很轻

如果组件只是：

- 几段文本
- 少量 JSX
- 简单布局
- 没有重计算

那它的渲染成本通常很低。

这时再加 `memo`，收益可能还不如一次浅比较的成本。尤其在现代 React 里，函数组件本身并没有你想象得那么重。

结论很直接：

- 轻组件，不要默认 `memo`
- 只有在你确认它是热点的时候，才考虑加

### 2. props 每次都在变成新引用

这是工程里最常见的失效场景。

```tsx
<MyComp onClick={() => doSomething(id)} />
<MyComp style={{ padding: 8 }} />
<MyComp items={[1, 2, 3]} />
```

表面上内容没变，实际上引用每次都变。

`React.memo` 默认做的是浅比较，不会深度理解对象内容，所以这种写法基本会让它失效。最后的结果通常是：

- `memo` 没起作用
- 你又开始加 `useMemo`、`useCallback`
- 组件和父组件都被性能优化逻辑污染

如果只是为了让 `memo` 生效，去制造更多缓存层，一般不值得。

### 3. 组件本来就会因为自己的 state 或 context 更新

很多人把 `memo` 误解成“父组件不变，子组件就不会更新”。

这不对。

组件自己有这些更新源时，`memo` 作用会明显变小：

- `useState`
- `useReducer`
- `useContext`
- 外部 store 订阅
- React Query、Redux、Zustand 这类数据源

尤其是 `useContext`。只要 context value 变了，消费它的组件通常还是会重新计算。

如果一个组件的主要更新来源根本不是父 props，而是内部状态或外部订阅，那它不是 `memo` 的理想对象。

### 4. 你还没有确认它真的是热点

这是最重要的工程原则。

不要因为“看起来会重渲染”就上 `memo`，要先看它是不是实际瓶颈。

很多时候真正耗时的不是这个子组件，而是：

- 父组件里复杂的数据处理
- 列表映射
- 某个昂贵 hook
- 大量上下文广播
- 虚拟列表没做好

如果热点不在这里，`memo` 基本不会改变整体性能。

### 5. 为了让 `memo` 生效，代码被迫变复杂

这个判断很简单：

- 如果你加了 `memo`
- 然后又被迫给一堆函数加 `useCallback`
- 给一堆对象加 `useMemo`
- 还要手动维护依赖关系

那通常说明你不是在优化，而是在迁就一个不合适的优化点。

这类代码常见后果是：

- 可读性下降
- 依赖关系变脆
- 后续改动容易引入 bug
- 性能收益却不稳定

工程上，复杂度本身就是成本。

### 6. 这是架构问题，不是局部渲染问题

如果状态放得太高、props 传得太深、context 粒度太粗、一次更新波及太多子树，那么 `memo` 只是补丁，不是根治。

这时候更值得做的通常是：

- 下放状态
- 缩小上下文范围
- 拆分组件边界
- 减少不必要的 props 透传

如果结构本身有问题，`memo` 只是延缓症状。

## 四、哪些场景下 `memo` 也许值得用

为了避免只讲“不该用”，也补一句正面判断。

`memo` 更适合这种组合：

- 组件渲染确实贵
- 父组件经常重渲染
- 组件的 props 大多稳定
- 组件不依赖频繁变化的 context

典型例子可能是：

- 复杂图表子组件
- 大表格里的重型 item
- 不常变但渲染代价高的区域

这类场景里，`memo` 才有机会把“重复渲染”变成“可跳过工作”。

## 五、工程里怎么判断要不要用

我通常只看三个问题：

1. 它是不是实际热点？
2. 它的 props 稳不稳定？
3. 它的渲染代价够不够高？

三个问题里，只要有一个答案是否定的，就先别急着上 `memo`。

如果要更具体一点，可以用这个顺序判断：

1. 先用 React DevTools Profiler 看真实耗时
2. 再看这次重渲染是不是由父组件连带引起的
3. 再看 props 能不能稳定下来
4. 最后才考虑 `memo`

这个顺序很重要，因为 `memo` 不是第一层优化手段，它是最后一层局部手段。

## 六、`memo`、`useMemo`、`useCallback` 不要绑成一套默认模板

很多问题其实不是 `memo` 单独造成的，而是它带出了一整套“缓存优先”的写法。

正确的理解应该是：

- `memo` 负责跳过组件函数执行
- `useMemo` 负责缓存计算结果
- `useCallback` 负责稳定函数引用

它们相关，但不是捆绑销售。

如果组件本身不重、props 也不稳定，那你为了配合 `memo` 去制造一堆缓存，常常是在用更高的复杂度换一个不确定的收益。

另外，如果项目已经在使用 React Compiler，很多手写优化的必要性会进一步下降。这个时候更应该先看编译器是否已经覆盖，再决定要不要保留手写 `memo`。

## 七、底层视角下最核心的一句话

`memo` 优化的是“重复计算”，不是“重复变化”。

如果一个组件本来就是变化源头：

- 输入真的变了
- state 真的变了
- context 真的变了

那 `memo` 不该也不可能阻止它更新。

它只适合切断一种情况：

- 父组件因为别的原因反复重渲染
- 但这个子组件实际上没有变化
- 于是它被连带执行了很多次

这才是 `memo` 的真正适用面。

## 结论

不要把 `memo` 当成默认配置。

我会倾向于不用 `memo` 的情况包括：

- 组件很轻
- props 经常变成新引用
- 组件本身就会因为 state 或 context 更新
- 你还没确认它是性能热点
- 为了让它生效，代码复杂度明显上升
- 本质问题是架构切分不合理

更实际的判断标准是：

- 渲染贵、props 稳、确实频繁重渲染，再考虑 `memo`
- 渲染便宜、props 不稳、或者更新本来就合理，就别用

`memo` 是一个局部工具，不是默认答案。

如果一段代码需要很多缓存才能看起来“优化成功”，通常说明你优化错了层。
